**One-paragraph explanation:**

> Before Nurye's explainer, [`eval_dev.py`](../../sales-agent-evaluation-bench/training/eval_dev.py) treated the 50-pair dev partition as a renewable resource — every checkpoint, hyperparameter sweep, and seed comparison ran the same evaluation against the same held-out set, with no record of how many times the partition had been accessed. The `MODEL_CARD.md` headline of 86.0% dev accuracy was conditional on ~27 such accesses, and I had no instrumentation to tell whether that count had crossed the contamination threshold the adaptive-data-analysis literature warns about. After Nurye's explainer (a) named the contamination rate as roughly proportional to `log(n_configs_compared)` and (b) pointed at the Reusable Holdout sample-complexity bound — for N_dev = 50 the practical ceiling is ~50 evaluations before the held-out estimate stops behaving as held-out — I made three coordinated changes. **First:** I added a `dev_access_log` to [`eval_dev.py`](../../sales-agent-evaluation-bench/training/eval_dev.py) that appends a row (timestamp, caller, config hash, observed accuracy) to a versioned JSONL on every call, so the access count is auditable and the trajectory of the metric across accesses is visible. **Second:** I added a Thresholdout-style cap — when `dev_access_log.count > 30`, the evaluator refuses to return a fresh number unless the new accuracy is within `tolerance = 0.02` of the cached estimate from the previous access for that config family, in which case it returns the cached number with a `cached: true` flag (this preserves the held-out's honesty for the *one number* I most need to track, the SimPO-vs-baseline lift, while still permitting bounded re-access). **Third:** I created [`eval_procedure.md`](../../sales-agent-evaluation-bench/training/eval_procedure.md) which (i) freezes the metric definition (preference accuracy on the 50-pair dev partition with the seed-median adapter, after the next retrain), (ii) carves a 12-pair untouched test partition out of the current dev set, leaving 38 dev pairs for ongoing selection — the explainer's tipping-point math justified eating the resolution cost now rather than waiting for V0.2 — and (iii) names the test partition as single-touch, refusing further evaluation against it once the V1.1 model card is written. The `MODEL_CARD.md` Reliability section now reports `dev_access_count = 27` alongside the 86.0% headline and a one-line caveat that this count is at the upper end of where the adaptive-data-analysis literature considers a held-out estimate reliable; the next training run will report against the locked test partition for the headline number with the dev partition relegated to selection-only. The decision-memo paragraph that referenced the bench's evaluation methodology will be revised in a follow-up commit; the seed-variance question separately flagged in [`question.md`](question.md) remains open as adjacent (see [`signoff.md`](signoff.md) — that is a different question for a different day).
